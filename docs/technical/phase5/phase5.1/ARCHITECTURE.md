# Phase 5.1: Robust Character Coordination & Conversational Routing - Architecture

**Version:** 1.0
**Last Updated:** February 20, 2026
**Status:** Draft
**Related PRD:** [PRD.md](PRD.md)

---

## Overview

This document describes the technical architecture for Phase 5.1. The redesign replaces the current three-component pipeline (IntentDetector → CharacterPlanner → fixed execution) with a **conversation-aware orchestration loop** that uses the LLM as a dynamic router and gives characters an explicit structured mechanism to signal handoffs.

---

## Current Architecture — Weaknesses

Before describing what gets built, it helps to map exactly where the current pipeline breaks for the target scenario.

```
Current Pipeline (per user message):

User Message
    │
    ▼
IntentDetector.detect()
  ├── _has_multi_task_indicators()  [regex on current message only]
  ├── _parse_multi_task_llm()       [context-free LLM call]
  └── _apply_rules()                [keyword patterns, no history]
    │
    ▼
CharacterPlanner.create_plan()
  ├── Maps intent categories → static assignment table
  ├── Marks sub-tasks is_dependent=True if LLM said so
  └── Moves dependent tasks to deferred_tasks list
    │
    ▼
ConversationManager
  ├── IF deferred_tasks present AND is_affirmation(message): _execute_deferred_tasks()
  │     [is_affirmation() = regex on message, no history]
  └── IF multi-character plan: _handle_phase45_multi_character()
        ├── Runs each character sequentially
        └── Adds handoff PROSE between characters (not functional second call)
                                    ▲
                  ╔══════════════════╩══════════════╗
                  ║  PROBLEM: Delilah's "invite      ║
                  ║  Hank" text appears but Hank     ║
                  ║  never executes his LLM call     ║
                  ╚═════════════════════════════════╝
```

**The core failure:** Handoffs are text generated by Delilah's LLM call. The orchestrator does not intercept them and run a real second character. Hank's execution only happens in `_handle_phase45_multi_character()` — which is only reached if the intent was classified as `multi_task` **at the first turn**, before the conversation has developed. By the time the user confirms Delilah's proposal (turn 2 or 3), the plan is already gone.

## Component Details

### Component 1: TurnClassifier

**File:** `backend/src/core/turn_classifier.py`

**Purpose:** Quickly determine the user's conversational intent relative to the current coordination state before deciding how to route.

**Interface:**

```python
from enum import Enum
from dataclasses import dataclass
from typing import Optional

class TurnType(Enum):
    AFFIRMATION = "affirmation"   # User is confirming/accepting a prior proposal
    NEW_REQUEST = "new_request"   # User is starting a new topic or task
    CLARIFICATION = "clarification"  # User is adding detail to the current topic
    REJECTION = "rejection"       # User is declining or cancelling a prior proposal

@dataclass
class TurnClassification:
    turn_type: TurnType
    confidence: float
    reasoning: str  # Short string for logging

class TurnClassifier:
    def classify(
        self,
        user_message: str,
        recent_history: List[Dict],        # Last 2 turns (user+assistant pairs)
        coordination_state: Optional[Dict] = None,  # Current pending state
    ) -> TurnClassification:
        ...
```

**Classification Strategy:**

The classifier uses a single short LLM call (not a regex). This is important: the regex `is_affirmation()` fails on content-carrying affirmations like "Southern Fried Chicken sounds great" and direct follow-on requests like "Can you add those to my list?"

```
System prompt: |
  You are a turn-classifier for a voice assistant multi-character system.
  Given the recent conversation and what is currently pending, classify
  the user's latest message as one of:
    - affirmation: user is saying yes/confirming/accepting the prior proposal
    - new_request: user is asking for something unrelated to current topic
    - clarification: user is adding context that refines the current topic
    - rejection: user is declining or cancelling

  Pending context (if any): {coordination_state.proposed_summary}

  Respond with JSON: {"type": "affirmation", "confidence": 0.92, "reasoning": "..."}

Temperature: 0.0   Max tokens: 80
```

**When to call TurnClassifier:**
- Only when `coordination_state.mode != "idle"` (there is something pending)
- When idle, skip the classifier and go directly to the ConversationRouter

**Fallback behavior:** If the LLM call fails, fall back to the existing regex `is_affirmation()` check for safety.

### Component 2: ConversationRouter

**File:** `backend/src/core/conversation_router.py`

**Purpose:** Given full recent conversation history and the new user message, decide which character should respond and whether a follow-up character action is needed after the primary character responds.

**Interface:**

```python
from dataclasses import dataclass, field
from typing import Optional, List

@dataclass
class PendingFollowup:
    character: str               # "hank"
    task_summary: str            # "Add the meal items Delilah described to the Sunday list"
    source: str                  # "router" | "handoff_tool"
    items: List[str] = field(default_factory=list)  # Optional structured list

@dataclass
class RoutingDecision:
    primary_character: str       # "delilah"
    pending_followup: Optional[PendingFollowup] = None
    rationale: str = ""          # Short string for logging

class ConversationRouter:
    def route(
        self,
        user_message: str,
        recent_history: List[Dict],     # Last 4-6 turns
        available_characters: List[str],
        coordination_state: Optional[Dict] = None,
    ) -> RoutingDecision:
        ...
```

**Router LLM Prompt Structure:**

```
System: |
  You are the routing coordinator for a multi-character voice assistant.
  Available characters and their domains:
    - delilah: cooking, recipes, meal planning, food advice, timers for food
    - hank: shopping lists, task management, reminders, scheduling, practical logistics

  Given the conversation so far and the user's latest message, decide:
    1. Which character should respond first (primary_character)
    2. Whether a second character should act AFTER the primary responds
       (e.g., Delilah describes a meal → Hank should add items to a list)

  A pending_followup is appropriate when:
    - The primary character's response will produce content that another character needs to act on
    - The user has made a request that spans two domains in a single message
    - The user's message is a direct request for the second character's domain

  Respond with JSON:
  {
    "primary_character": "delilah",
    "pending_followup": {           // omit if not needed
      "character": "hank",
      "task_summary": "Add the meal items from Delilah's response to the Sunday Meal shopping list"
    },
    "rationale": "User asking for meal plan + list; Delilah plans, Hank logs"
  }

Temperature: 0.1   Max tokens: 150
```

**Routing Decision Matrix (for reference — the LLM infers this natively):**

| User Message Pattern | Primary | Followup |
|---|---|---|
| "Set a timer for X" | delilah | none |
| "Add X to my list" | hank | none |
| "Set timer AND add to list" | delilah | hank (add to list) |
| "Plan a meal" | delilah | none (no list yet) |
| "Plan a meal and make a shopping list" | delilah | hank (add items after Delilah proposes) |
| "Can you add those items to my list?" (after recipe) | hank | none |
| "What's the weather?" | rex (future) / delilah (now) | none |

### Component 3: ConversationStateManager

**File:** `backend/src/core/conversation_state.py`

**Purpose:** Manage the coordination state stored in `ConversationContext.metadata`. Replaces the ad-hoc `deferred_tasks` / `deferred_tasks_expires_at` / `_phase45_turn_character` keys with a typed, queryable state object.

**State Schema:**

```python
from enum import Enum
from dataclasses import dataclass, field
from typing import Optional, List
from datetime import datetime

class CoordinationMode(Enum):
    IDLE = "idle"
    PROPOSING = "proposing"          # Primary character has made a proposal; followup pending
    AWAITING_ACTION = "awaiting_action"  # User confirmed; secondary character action pending

@dataclass
class CoordinationState:
    mode: CoordinationMode = CoordinationMode.IDLE
    pending_character: Optional[str] = None      # "hank"
    pending_task: Optional[str] = None           # "Add Southern Fried Chicken items to Sunday list"
    pending_items: List[str] = field(default_factory=list)  # Extracted structured items
    proposed_summary: Optional[str] = None       # Free-text summary of what primary character proposed
    expires_at: Optional[datetime] = None        # Auto-clear after 20 min
    last_updated: Optional[datetime] = None

class ConversationStateManager:
    def get_state(self, context: ConversationContext) -> CoordinationState: ...
    def set_proposing(self, context, pending_character, pending_task, proposed_summary, items=None): ...
    def set_awaiting_action(self, context, pending_character, pending_task, items=None): ...
    def clear(self, context): ...
    def is_expired(self, state: CoordinationState) -> bool: ...
```

**Key transitions:**

```
IDLE
  →  PROPOSING     when router returns a pending_followup (after primary executes)
  →  IDLE          when no followup needed

PROPOSING
  →  AWAITING_ACTION  when TurnClassifier returns AFFIRMATION
  →  IDLE             when TurnClassifier returns REJECTION or NEW_REQUEST
  →  IDLE             when state expires (20 min)

AWAITING_ACTION
  →  IDLE             after pending character executes successfully
  →  IDLE             on any failure
```

**Content extraction after primary response:**

When the router sets a pending followup, the orchestrator runs a short extraction pass after the primary character's response to populate `pending_items`. This uses a lightweight LLM call:

```
System: Extract a structured list of items from this response that should be added
to a shopping list. Return JSON: {"items": ["whole chicken", "buttermilk", ...]}
If no clear list, return {"items": []}.
Max tokens: 100
```

This decouples "what Delilah said" from "what Hank needs to do" — solving the context leakage problem where Hank previously had to re-read Delilah's entire response.

### Component 4: `request_handoff` Tool

**File:** `backend/src/tools/handoff_tool.py`

**Purpose:** Give characters a first-class mechanism to signal they want to hand off to another character. This is an explicit tool call that the orchestrator intercepts — not prose that the orchestrator guesses at.

**Tool Definition (OpenAI function format):**

```json
{
  "name": "request_handoff",
  "description": "Signal that you have completed your part of the task and would like another character to handle a follow-up action. Call this when you have finished dealing with your domain and there's clearly remaining work for a different character.",
  "parameters": {
    "type": "object",
    "properties": {
      "to_character": {
        "type": "string",
        "enum": ["hank", "rex"],
        "description": "The character who should handle the follow-up"
      },
      "task_summary": {
        "type": "string",
        "description": "Brief description of what the other character should do (1-2 sentences)"
      },
      "items": {
        "type": "array",
        "items": {"type": "string"},
        "description": "Optional structured list of items to pass to the next character (e.g., ingredients)"
      }
    },
    "required": ["to_character", "task_summary"]
  }
}
```

**Orchestrator interception logic (in CharacterExecutor):**

```python
# After primary character LLM response loop completes:
handoff_signal = None
for tool_call in all_tool_calls:
    if tool_call["function"]["name"] == "request_handoff":
        handoff_signal = json.loads(tool_call["function"]["arguments"])
        break  # Only handle first handoff per turn

if handoff_signal:
    # Do NOT add a tool result for request_handoff — it has no output
    # Store it for the ResponseAssembler to trigger secondary execution
    ...
```

**Character prompt guidance:**

Characters are told in their system prompts:
> You have a special tool: `request_handoff`. Use it when you have finished your part
> of a task and a follow-up action belongs to another character. For example, after you
> suggest a recipe, if the user has a shopping list in mind, use `request_handoff` to
> let Hank build the list. You may speak your handoff naturally alongside the tool call —
> your spoken words will appear in the response. Do NOT use `request_handoff` when the
> full task is within your own domain.

**Precedence rule:** If both the router's `pending_followup` AND a `request_handoff` tool call exist, the tool call takes precedence (it carries more specific context from the character itself).

### Component 5: CharacterExecutor (Refactored)

**File:** `backend/src/core/character_executor.py` (new; extracts logic from `conversation_manager.py`)

**Purpose:** Run a single character's LLM call with the full tool loop. Returns both the text response and any handoff signal detected.

**Interface:**

```python
@dataclass
class CharacterResponse:
    character: str
    text: str
    voice_mode: str
    tool_calls_made: int
    handoff_signal: Optional[dict] = None  # Populated if request_handoff was called

class CharacterExecutor:
    async def execute(
        self,
        character: str,
        task_description: str,
        context: ConversationContext,
        session_id: str,
        user_id: str,
        system_prompt_override: Optional[str] = None,
    ) -> CharacterResponse:
        ...
```

**Why extract this from ConversationManager:**

`ConversationManager._handle_phase45_multi_character()` currently contains ~150 lines that duplicate the single-character execution loop. Extracting `CharacterExecutor` gives a single, tested, reusable unit for any character execution — primary, secondary, deferred, or direct.

## Data Flow: Two-Turn Meal Plan → Shopping List

### Turn 1: "Can you help me plan a meal for Sunday then make me a shopping list?"

```
1. TurnClassifier: state=IDLE → skip (go direct to router)

2. ConversationRouter:
   Input: user message + last 2 turns (empty first time)
   Output: RoutingDecision {
     primary_character: "delilah",
     pending_followup: {
       character: "hank",
       task_summary: "After Delilah proposes the meal, add the ingredients to Sunday Meal list"
     }
   }

3. CharacterExecutor: run delilah
   - System prompt: Delilah's cooking persona
   - Response: "What flavors are you in the mood for? Something Southern?"
   - No request_handoff tool call (not enough info yet)

4. ConversationStateManager.set_proposing():
   mode: PROPOSING
   pending_character: "hank"
   pending_task: "After Delilah proposes the meal, add ingredients to Sunday Meal list"
   proposed_summary: (empty at this point — Delilah hasn't proposed yet)
   expires_at: now + 20 min

5. Response to user: Delilah's question
```

### Turn 2: "Southern Fried Chicken sounds great. No dietary restrictions."

```
1. TurnClassifier: state=PROPOSING
   Input: user message + last 1 turn (Delilah's question)
   + proposed_summary: (empty)
   Output: TurnClassification { type: CLARIFICATION, confidence: 0.91 }
   → mode stays PROPOSING (user gave Delilah the info she needs, not a final confirmation)

2. ConversationRouter: NOT called (turn is CLARIFICATION, we route to primary character)
   → primary continues to be "delilah"

3. CharacterExecutor: run delilah (continuation)
   - Delilah now has the choice. She may:
     a) Propose the menu and call request_handoff(to="hank", task="add ingredients", items=[...])
     b) Propose the menu without a handoff (if user hasn't confirmed yet)
   - In this case: Delilah provides the complete menu + calls request_handoff
   - request_handoff detected by orchestrator

4. Item extraction: extract items from Delilah's response
   → ["whole chicken (3-4 lbs)", "buttermilk (2 cups)", ...]

5. SecondCharacterExecutor: run hank
   - task: "Add these Southern Fried Chicken ingredients to the Sunday Meal shopping list: [items]"
   - Hank calls add_items_to_list tool
   - Hank responds: "Aye, Cap'n. Got 'em on the list — chickens, buttermilk, the lot."

6. ConversationStateManager.clear()

7. ResponseAssembler:
   fragments: [
     { character: "delilah", text: "Alright darlin'! Here's your menu... [handoff phrase]", voice_mode: "passionate" },
     { character: "hank", text: "Aye, Cap'n. List is set.", voice_mode: "working" }
   ]
```

### Turn 3 (alternate path): "Can you add those items to my list?"

```
1. TurnClassifier: state=PROPOSING
   Input: user message + Delilah's menu response
   Output: TurnClassification { type: AFFIRMATION, confidence: 0.88 }
   → OR the router detects this as a direct LIST request regardless of state

2. ConversationRouter: detects "add those items to my list" → primary_character = "hank"
   Context: recent history has Delilah's recipe list — Hank can extract from there

3. CharacterExecutor: run hank directly
   - Hank reads conversation history, finds Delilah's list
   - Calls add_items_to_list tool
   - Responds: "Aye, Cap'n. Added 'em all."

4. ConversationStateManager.clear()
```

## Modified Execution Path in ConversationManager

The `handle_user_message` method is refactored to use the new components:

```python
async def handle_user_message(self, session_id, user_message, user_id, input_mode):
    context = self.get_or_create_conversation(session_id, user_id)
    # ... save message to history ...

    # Step 1: Get current coordination state
    coord_state = self.state_manager.get_state(context)

    # Step 2: Clear expired state silently
    if coord_state.mode != CoordinationMode.IDLE and self.state_manager.is_expired(coord_state):
        self.state_manager.clear(context)
        coord_state = self.state_manager.get_state(context)

    # Step 3: Classify turn type if something is pending
    turn_type = TurnType.NEW_REQUEST
    if coord_state.mode != CoordinationMode.IDLE:
        classification = self.turn_classifier.classify(
            user_message, context.history[-4:], coord_state
        )
        turn_type = classification.turn_type

    # Step 4: Handle affirmation of pending action
    if turn_type == TurnType.AFFIRMATION and coord_state.mode == CoordinationMode.PROPOSING:
        self.state_manager.set_awaiting_action(...)  # transition state
        return await self._execute_pending_character(...)

    # Step 5: Clear state on rejection or topic change
    if turn_type in (TurnType.REJECTION, TurnType.NEW_REQUEST):
        self.state_manager.clear(context)

    # Step 6: Route to primary character
    routing = self.router.route(
        user_message,
        context.history[-6:],
        available_characters=self.available_characters,
    )

    # Step 7: Execute primary character
    primary_response = await self.executor.execute(
        character=routing.primary_character,
        task_description=user_message,
        context=context,
        session_id=session_id,
        user_id=user_id,
    )

    # Step 8: Determine if secondary execution is needed
    followup = primary_response.handoff_signal or (
        routing.pending_followup.__dict__ if routing.pending_followup else None
    )

    if followup:
        # Extract proposed content (items list) from primary response
        proposed_items = await self._extract_items(primary_response.text)

        secondary_response = await self.executor.execute(
            character=followup["character"],
            task_description=self._build_secondary_task(followup, proposed_items),
            context=context,
            ...
        )
        self.state_manager.clear(context)
        return self._assemble_response([primary_response, secondary_response])
    else:
        # Store pending followup from router for next turn (if applicable)
        if routing.pending_followup:
            self.state_manager.set_proposing(
                context,
                pending_character=routing.pending_followup.character,
                pending_task=routing.pending_followup.task_summary,
                proposed_summary="",  # Will be populated after primary responds
            )
        return self._assemble_response([primary_response])
```

## LLM Call Budget per Turn

| Turn Type | LLM Calls | Notes |
|---|---|---|
| Idle, single-character | 2 | Router (fast) + Character |
| Idle, with followup | 3–4 | Router + Primary + Item extraction + Secondary |
| Pending state, affirmation | 2–3 | Classifier (fast) + Secondary + optional extraction |
| Pending state, new request | 2 | Classifier (fast) + Router + Character |
| Character uses request_handoff | 2–3 | Character (includes tool call) + Secondary + optional extraction |

This is comparable to the current Phase 4.5 multi-character flow (which also runs 2+ LLM calls for multi-character turns) but adds the TurnClassifier call — which is bounded at `max_tokens=80` for minimal latency impact.

---

## Fallback & Error Handling

| Failure | Fallback |
|---|---|
| ConversationRouter LLM call fails | Route to `delilah` (existing default); log ERROR |
| TurnClassifier LLM call fails | Fall back to regex `is_affirmation()` from Phase 5; log WARNING |
| Secondary character execution fails | Return primary response only; clear coordination state; log ERROR |
| `request_handoff` to unavailable character | Log WARNING; treat as single-character response |
| Item extraction fails (empty list) | Pass `task_summary` only to secondary character (no items list) |

### request_handoff Tool Call Logging

`request_handoff` is logged as a first-class tool call in the existing tool calls store, but with an extended schema to surface coordination-specific context.

**Log Entry Shape:**

```typescript
interface HandoffToolCallLog extends ToolCallLog {
  tool_name: "request_handoff";
  character: string;           // who initiated — "delilah"
  arguments: {
    to: string;                // target character — "hank"
    task_summary: string;      // what Delilah is handing over
    items: string[];           // structured items if provided
  };
  orchestrator_decision: {
    accepted: boolean;         // did the orchestrator honour it?
    rejected_reason?: string;  // e.g. "second handoff in turn — FR2.4"
  };
  resulted_in_character: string | null; // who actually ran next
}
```

**What this surfaces in the UI:**

- The Tool Calls tab shows `request_handoff` inline with other tool calls in turn order, so you can see exactly when in the conversation Delilah made the request
- `orchestrator_decision.accepted` makes FR2.4 violations visible — if a second handoff was blocked you'll see `accepted: false` with the reason rather than silently dropping it
- `resulted_in_character` closes the loop — you can confirm Hank actually ran, not just that Delilah asked

**Why this matters for debugging:**

The original handoff bug was invisible in the logs — Delilah produced prose saying she'd "invite Hank" but no tool call ever fired. With `request_handoff` as a real tool call, a broken handoff will show up as one of three distinct failure modes:

1. Tool call is **missing entirely** — Delilah's prompt isn't triggering it
2. Tool call is present but `accepted: false` — the orchestrator blocked it (FR2.4 or unavailable character)
3. Tool call is present and accepted, but `resulted_in_character` is null — orchestrator bug

Each failure mode points directly at a different fix.
